## 1. Problem Statement

D&M is replacing its phone system (Microsoft Teams → **Allo**). Allo performs caller-ID
lookup against **Attio** (D&M's own CRM) and screen-pops the matched contact to the CSO
(Clarissa Aurisch) when a call comes in. Today the CSO answers blind — no idea who is
calling, what they've booked, or whether they have an open issue.

We want Verco data projected into Attio so that when a Kwinana resident calls, the
screen-pop carries **full context**: who they are, their booking history, open tickets,
their collection area — with a one-click link into live Verco to act.

## 2. Goals

- **(Phase 1)** A Kwinana resident's inbound call resolves (via phone match) to an Attio
  person carrying a Verco context summary attribute + a deep-link to the live Verco admin
  contact page. (Pop-visibility of the summary attribute is verified live; lists are the
  fallback glance — see §0.)
- **(Phase 1.5, gated)** Bookings and service tickets appear in Attio as **read-only**
  records associated to the person, so the CSO can see/click individual ones inside Attio.
  Requires Attio Pro + the live pop test (§0).
- Verco remains the single source of truth. The CSO acts in Verco (via deep-link), never
  writes resident data back from Attio (this phase).
- Smallest thing that handles a real Kwinana call. Learn from live usage, then widen.

## 3. Non-Goals (this phase)

- **All clients except Kwinana** — Verge Valet/WMRC, Rockingham, Karratha contacts do NOT
  sync yet. (WMRC has its own parallel HubSpot feed; Clarissa's WMRC call volume is being
  confirmed before deciding whether VV joins the Attio feed.)
- **Two-way actions** (cancel/close/respond from Attio → Verco) — on the roadmap, designed
  for forward-compatibility, not built now.
- **Real-time push** — minutes-fresh is acceptable (see §4 rationale); no DB triggers /
  event queue this phase.
- **The generic webhook foundation** from the HubSpot spec — deliberately deferred until a
  second consumer (WMRC HubSpot) is a confirmed co-build (see §6 Approach B).
- **Allo configuration** — the Allo↔Attio caller-ID link is Allo-side config (confirmed
  native, withallo.com), not Verco code. We build Verco → Attio only.
- **Custom objects + Pro tier (Phase 1)** — `verco_booking`/`verco_ticket` and the Attio Pro
  upgrade are deferred to Phase 1.5, gated on the live pop test (§0). Phase 1 ships on Plus.

## 4. Chosen Approach — Batch cron watermark sync (Approach A)

A single `sync-to-attio` Edge Function, invoked by pg_cron every ~3 minutes:

1. Read the per-entity cursor `(last_synced_at, last_synced_id)` from `attio_sync_state`.
2. SELECT up to **N rows** (self-limit, e.g. 200) of Kwinana contacts / bookings /
   service_tickets changed since the cursor, `WHERE (updated_at, id) > (last_synced_at,
   last_synced_id)` ordered by `(updated_at, id)`. The compound cursor avoids skipping rows
   that share an `updated_at` (see §6.5).
3. **Phase 1:** upsert each affected **person** into Attio via the REST API, matched on the
   **cached Attio `record_id`** (`contacts.attio_person_id`): NULL → `POST` create + capture
   id; set → update by id. (Attio doesn't allow a custom unique attribute on People, §11.2, so
   `verco_contact_id` can't be the key.) **Conditionally** write the returned record id + web
   URL back into `contacts.attio_person_id` / `attio_person_web_url` (conditional to avoid the
   loop — §6.2). Bookings/tickets are read for the summary but **not written to Attio** in
   Phase 1. (Phase 1.5 adds `verco_booking`/`verco_ticket` object upserts keyed on
   `verco_booking_id`/`verco_ticket_id`, writing back to `service_ticket.attio_record_id` +
   `booking.attio_record_id`.)
4. Recompute and push `verco_context_summary` for every person whose contact/booking/ticket
   changed this run (the affected-person set — see §7). A booking/ticket change drives a
   **person** update (summary), not a separate-object write, in Phase 1.
5. Advance the cursor to the `(updated_at, id)` of the last successfully-upserted row —
   **only after** its Attio write succeeds. A mid-batch failure leaves the cursor at the
   last good row; the next run resumes there. Idempotent upsert makes re-processing the
   boundary row harmless.
6. Return HTTP 500 to pg_cron if any row fails (so a partial failure is visible, per
   CLAUDE.md §11 — never hide partial failure behind a 200). On Attio 429, back off and
   return 500 so the next cron tick retries from the cursor.
7. Because each run is self-limited to N rows and the cursor persists, the */3 cron
   naturally **drains a backlog** (including the initial backfill) across successive ticks
   without any one invocation risking the EF wall-clock limit.

```
pg_cron (*/3 * * * *)
      │
      ▼
sync-to-attio EF  (service role — batch admin op, documented)
      │  reads attio_sync_state compound cursor (updated_at, id)
      ├── SELECT contacts WHERE EXISTS(KWN booking) AND (updated_at,id) > cursor  LIMIT N
      ├── SELECT booking  WHERE client_id = KWN AND (updated_at,id) > cursor      LIMIT N
      ├── SELECT service_ticket WHERE client_id = KWN AND (updated_at,id) > cursor LIMIT N
      │
      ▼  Attio REST API (PUT assert/upsert, bearer token)
   People (+ custom attrs)   verco_booking object   verco_ticket object
      │
      ▼  advance attio_sync_state.last_synced_at per entity
```

### 4.1 Why batch cron, not event-driven (the key design call)

- **The CSO never feels the lag.** The screen-pop's value is identity + a glanceable
  summary; the deep-link always opens *live* Verco. A 3-minute-stale Attio copy is invisible
  in practice — a resident who booked online 90 seconds ago is unlikely to also be on the
  phone in that window, and if they are, the deep-link shows truth.
- **Fewest moving parts** — no triggers, no per-row queue, no dead-letter table. Replay is
  trivial: reset the watermark and re-run.
- **Strict subset of the event-driven foundation.** The Attio-side schema (§5) and summary
  logic (§7) are exactly what the shared webhook foundation (Approach B) would need. If
  volume later demands real-time, or WMRC HubSpot becomes a confirmed co-build, graduating
  A → B is a contained transport swap, not a rewrite.
- **Don't build the generic event bus for a handful of Kwinana calls a day.** Earn the
  abstraction with a second real consumer.

## 5. Attio-Side Schema (one-time setup)

Built once; reused unchanged if the sync later graduates to event-driven.

### 5.1 People (standard object) — add custom attributes

| Attribute (api_slug) | Type | Notes |
|---|---|---|
| `verco_contact_id` | text, **NON-unique** | Verco `contacts.id` — reference/search only. **NOT the upsert key** — Attio does not allow custom unique attributes on People (verified §11.2). |
| `verco_context_summary` | text | Glanceable string, recomputed on any related change (§7) |
| `verco_url` | text | Deep-link `https://verco.au/admin/contacts/{id}` |
| `service_area` | text (or select) | e.g. `KWN` — segment now, filter/expand later |

**Upsert mechanism for People (no custom unique attribute available):** match on the cached
Attio `record_id`, not on `verco_contact_id`.
- `contacts.attio_person_id` **NULL** → `POST` create the person → capture the returned
  `record_id` → conditional-writeback into `contacts.attio_person_id` (§6.2, eng-review
  Issue 1).
- `contacts.attio_person_id` **set** → update that record by id (`PATCH`/`PUT .../records/{id}`).
The advisory lock (Issue 2) guarantees no two runs both see NULL and double-create. (Custom
objects in Phase 1.5 — `verco_booking`/`verco_ticket` — *do* support unique attributes, so
they upsert by `verco_booking_id`/`verco_ticket_id`; only People needed this change.)

- `phone_numbers` (standard, E.164) is the **Allo match key** — populated from
  `contacts.mobile_e164` (nullable; a contact with no mobile won't match an inbound call).
  Note it is `is_unique: false`, so it is NOT the upsert key.
- `email_addresses` (standard, unique) populated from `contacts.email` (non-null in schema).
- `name` (standard personal-name) from `contacts.first_name` + `last_name` (both exist
  alongside the generated `full_name` — verified against `types.ts`).

**Reverse-link = the upsert key (already scaffolded in Verco — use it, don't reinvent):**
`contacts` already has `attio_person_id`, `attio_person_web_url`, `last_synced_by`. On create,
write the Attio record id → `attio_person_id`, web URL → `attio_person_web_url`, stamp
`last_synced_by`. `attio_person_id` is **both** the local cache of the Attio id (reverse
deep-linking, future two-way path) **and** the match pointer for subsequent upserts — Attio
won't let us match on a custom unique attribute on People, so the cached record_id is the key.
The conditional writeback (§6.2) is doubly load-bearing here.

### 5.2 New custom object `verco_booking` — PHASE 1.5 (gated: Pro + live pop test, §0)

`booking` has no date/price/address columns of its own — these come from joins. Verified
sourcing (against `types.ts`):

| Attribute | Source |
|---|---|
| `verco_booking_id` (unique) | `booking.id` |
| `ref` | `booking.ref` (e.g. KWN-12345) — column is `ref`, **not** `reference` |
| `status` | `booking.status` (enum `booking_status`) |
| `collection_date` | `MIN(collection_date.date)` via `booking_item.collection_date_id` → `collection_date.date` (a booking can span multiple items/dates; take the earliest as "scheduled collection") |
| `address` | `eligible_properties.formatted_address` ?? `eligible_properties.address` via `booking.property_id`; fallback `booking.geo_address` / `booking.location` |
| `services_summary` | aggregated from `booking_item` (`service.name` × `no_services`), e.g. "1× General Bulk, 2× Mattress" |
| `total` | `SUM(booking_item.unit_price_cents × booking_item.no_services)` → currency. There is **no** `booking.total_cents` — price lives on `booking_item` (CLAUDE.md §6) |
| `verco_url` | `https://verco.au/admin/bookings/{id}` |
| → person | record-reference (association) via `verco_contact_id` (`booking.contact_id`, nullable — skip unassociated bookings) |

**Reverse-link:** add `booking.attio_record_id text` in this migration (`booking` lacks one;
contacts + service_ticket already have theirs). Write the Attio booking record id back to it.

### 5.3 New custom object `verco_ticket` — PHASE 1.5 (gated: Pro + live pop test, §0)

| Attribute | Source |
|---|---|
| `verco_ticket_id` (unique) | `service_ticket.id` |

## 6. Verco-Side Changes

### 6.1 New table `attio_sync_state`

```sql
CREATE TABLE attio_sync_state (
  entity          text PRIMARY KEY CHECK (entity IN ('contact','booking','service_ticket')),
  last_synced_at  timestamptz NOT NULL DEFAULT '1970-01-01T00:00:00Z',
  last_synced_id  uuid        NOT NULL DEFAULT '00000000-0000-0000-0000-000000000000',
  last_run_at     timestamptz,
  last_run_status text,
  rows_synced     int NOT NULL DEFAULT 0
);
```

- Compound cursor `(last_synced_at, last_synced_id)` — not a bare timestamp — so rows sharing
  an `updated_at` are never skipped (see §6.5).
- RLS: `contractor-admin` SELECT only (observability). Writes happen via service role in the
  EF. No client/resident/field/ranger access.
- `booking.attio_record_id text` is a **Phase 1.5** addition (only needed once `verco_booking`
  objects are written back). Phase 1 doesn't write bookings to Attio, so it's not in the
  first migration. (`contacts.attio_person_id` + `service_ticket.attio_record_id` already
  exist; Phase 1 only uses `contacts.attio_person_id`.)
- **Keyset indexes (eng-review Issue 5):** add `CREATE INDEX ... ON {contacts,booking,
  service_ticket} (updated_at, id)`. The cursor reads all three (bookings/tickets drive the
  person-summary recompute even in Phase 1), ordered by `(updated_at, id)`; without these it
  seq-scans + sorts each table every tick. (`idx_booking_contact` already covers the scope
  EXISTS.)

### 6.2 New Edge Function `sync-to-attio`

- Service role (documented why: batch admin op, reads across the contact→booking graph,
  writes to an external CRM — per CLAUDE.md §11). **Never** invoked from `app/` code.
- Secrets (Supabase dashboard, never in `.env`): `ATTIO_API_KEY`, and optionally
  `ATTIO_KWN_CLIENT_ID` (the Kwinana `client.id`) to keep the scope filter config-driven.
- Pattern: **acquire `pg_try_advisory_lock`** → auth → read cursor → query ≤N rows per entity
  → for each person: `attio_person_id` NULL → create (capture id), else update by id →
  **conditionally** write the id back → recompute affected-person summaries → advance cursor →
  release lock. Returns 500 on any per-row failure or Attio 429. (Match is by cached
  `record_id`, not a custom unique attribute — §5.1/§11.2.)
- **Advisory lock (eng-review Issue 2):** `pg_try_advisory_lock(<const>)` at start; if not
  acquired, another run is in flight → log + return 200 no-op. Prevents two overlapping
  backfill runs from racing the single cursor row and skipping rows.
- **Conditional writeback (eng-review Issue 1 — CRITICAL):** write
  `attio_person_id`/`attio_person_web_url`/`last_synced_by` (Phase 1; `booking`/`ticket`
  `attio_record_id` in Phase 1.5) **only `WHERE attio_person_id IS DISTINCT FROM $new`**. The
  `handle_updated_at` trigger (`initial_schema.sql:85`) bumps `updated_at` on EVERY update —
  an unconditional writeback would re-bump `updated_at`, the cursor would re-select the row
  next tick, and it would re-sync forever. Conditional writeback converges after one cycle
  (first sync writes once → one extra re-select → value now stable → no write → no bump).
- **Pure logic placement (eng-review Issue 3):** mappers + `computeContextSummary()` +
  cursor-compare live in `src/lib/attio/` (Vitest, 100%), mirrored by the EF — same pattern
  as `src/lib/pricing/calculate.ts` ↔ `_shared/pricing.ts` (CLAUDE.md §6). The future HubSpot
  emitter reuses the same entity→DTO shaping.
- **Self-limited** to N rows/entity/run (e.g. 200) so a single invocation never approaches the
  EF wall-clock limit (~150s default / 400s max). Backlog and the initial backfill drain
  across successive */3 ticks (§4 step 7) — there is no "one big run".
- **Attio rate limits:** the steady-state */3 cadence is well within Attio's per-token limit,
  but the draining backfill could hit it. On 429, stop the batch, leave the cursor at the last
  good row, return 500; next tick resumes. Optional: small inter-request pace (HubSpot spec §8
  used 50/batch + 1s) if 429s appear.

### 6.3 Scope filter (Kwinana wedge)

- `booking` / `service_ticket`: `WHERE client_id = <KWN>`.
- `contacts`: include a contact with `EXISTS (SELECT 1 FROM booking b WHERE b.contact_id =
  c.id AND b.client_id = <KWN>)`. **Do NOT reuse** the audit trigger's contact→client
  derivation (`audit_trigger_derive_client_id_admin.sql:118`) — that takes the *single most
  recent* booking (`ORDER BY created_at DESC LIMIT 1`), so it would wrongly exclude a contact
  whose latest booking is non-Kwinana but who has an older Kwinana one. Membership is "has
  *any* Kwinana booking", which is an EXISTS, not a most-recent lookup. (Resolves the HubSpot
  spec's open question #6 — correctly this time.)

### 6.4 pg_cron schedule

`*/3 * * * *`. Wrap the `cron.schedule` migration in
`DO $$ IF EXISTS cron.unschedule $$ END` so it's re-appliable (CLAUDE.md §11). Invoke via
`net.http_post` with a service-role bearer — same mechanism proven in
`20260327120000_nightly_sync_cron.sql`.

### 6.5 Watermark boundary correctness

`handle_updated_at()` stamps `updated_at = now()` (transaction-start time), so multiple rows
can share an `updated_at` within a busy window. A bare `updated_at > wm` cursor can then skip
a row: if two rows share the max `updated_at` and only one is processed before the run ends,
the next run's strict `>` never revisits the other. Mitigations, combined:

1. **Compound cursor** `(updated_at, id)` with `WHERE (updated_at, id) > (last_synced_at,
   last_synced_id)` ordered by `(updated_at, id)` — total order, no ties.
2. **Advance only after a successful Attio upsert**, row by row (or after the batch, to the
   last good row).
3. **Idempotent upsert** — People update by cached `record_id` (custom objects by their unique
   key) — re-processing a boundary row is a no-op, so even at-least-once delivery is safe.

## 7. Context Summary Computation

`verco_context_summary` is a denormalised cache on the person, recomputed whenever any of
their contacts/bookings/tickets rows change within the batch. Proposed format:

```
KWN · {active_booking_count} bookings · next {next_collection_date} · {open_ticket_count} open ticket(s)
```

Example: `KWN · 2 bookings · next 02/06 · 1 open ticket`

- Computed server-side in the EF from the resident's Kwinana bookings + open tickets.
- Empty/zero states handled explicitly (`KWN · no active bookings`).
- This is the glance; the associated `verco_booking` / `verco_ticket` records + the
  deep-link carry the detail.

**Affected-person set (the fan-out):** a changed *booking* or *ticket* row resolves to its
owning `contact_id`; a changed *contact* row is itself. Union these into a distinct set of
contact ids for the run, then for each compute the summary in one query (their Kwinana
bookings + open ticket count). Bounded by N rows/run, so the fan-out is at most N contacts ×
one aggregate query each — cheap at Kwinana volume. The person upsert (summary) and the
booking/ticket upserts in the same run may briefly disagree until both are written; the
deep-link is always live, so the CSO never acts on a half-updated cache.

## 8. Phone Matching — Reality Check (premise P2)

Allo resolves a call only when the caller's number equals a `phone_numbers` value on an
Attio person. Match will NOT be 100%:

- Withheld / private numbers → no match.
- Landline or a partner's mobile when we hold a different `mobile_e164` → no match.
- Stale number → no match.

**Clarissa always needs a manual search fallback in Attio/Allo.** We're buying *most* calls
pre-identified, not all. Measure the match rate after go-live (§10).

## 9. Security & Data Governance

| Concern | Mitigation |
|---|---|
| Resident PII into a 3rd-party CRM | D&M is operator/processor for Kwinana; name/phone/email/booking summary in D&M's own Attio is within the existing data-handling relationship. CSO (full-PII role) is the consumer — field/ranger zero-PII rule does not apply. |
| Attio API key leak | Stored as EF secret (Supabase dashboard); never in repo or `.env`; rotatable. |
| Over-broad sync | Hard scope filter to Kwinana `client_id`; success criterion verifies zero non-Kwinana contacts in Attio. |
| Service role misuse | Confined to the one cron EF, documented; never reachable from `app/`. |
| Stale/incorrect context | Summary is a cache; deep-link always shows live Verco truth — the CSO acts on Verco, not the Attio copy. |

## 10. Success Criteria

- A Kwinana resident calls; Allo screen-pops the matched Attio person showing name +
  `verco_context_summary`; the `verco_url` deep-link opens the live Verco admin contact page.
- A Kwinana booking or ticket change is reflected in Attio within ~5 minutes.
- `verco_booking` / `verco_ticket` records appear in Attio associated to the right person,
  each with a working `verco_url`.
- **Zero** contacts/bookings/tickets from non-Kwinana clients in Attio (scope filter holds).
- Phone-match rate (Kwinana inbound calls that resolve to an Attio person) measured and
  baselined in the first month — informs whether to enrich phone data.
